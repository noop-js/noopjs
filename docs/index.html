NoopJS

NoopJS is a zero-runtime, resumable, signal-based web framework. It compiles your components into vanilla JavaScript that creates DOM nodes directly — no Virtual DOM, no diffing, no framework overhead on the client.

Core Principles

  • No Virtual DOM — The compiler generates direct DOM creation and fine-grained signal-driven updates.
  • Resumability, Not Hydration — Server-rendered pages resume on the client without re-running component code. The reactive graph is serialized and re-attached.
  • Zero Runtime — Pages with no interactivity ship zero framework JavaScript. Interactive islands ship only the minimal resumer (~1.5 KB) and exact handler code.
  • TC39 Signals — Fine-grained reactivity via signal, computed, and effect following the TC39 proposal semantics.
  • Platform Native — Uses browser primitives: Custom Elements, View Transitions API, Fetch, ReadableStream.
NoopJS is currently in early development. The APIs are stable but expect iteration as we approach 1.0.

Quick Start

Create a counter component (src/counter.noop.tsx):

import { signal } from '@noopjs/signals';

export const styles = {
  container: { padding: '16px', textAlign: 'center' },
  count:    { fontSize: '24px', fontWeight: 'bold' },
  button:   { padding: '8px 16px', cursor: 'pointer' },
};

export default function Counter() {
  const count = signal(0);
  return (
    <div className={styles.container}>
      <p className={styles.count}>{count}</p>
      <button className={styles.button} onClick={() => count.set(count.get() + 1)}>
        Increment
      </button>
    </div>
  );
}

Mount it (src/main.ts):

import Counter from './counter.noop';

const root = document.getElementById('root');
root!.appendChild(Counter());

Add the Vite plugin (vite.config.ts):

import { defineConfig } from 'vite';
import { noopVite } from '@noopjs/vite';

export default defineConfig({
  plugins: [noopVite()]
});

Start dev server:

npx vite
The .noop.tsx extension tells the plugin to compile this file. Standard .tsx files are left untouched.

Signals

NoopJS uses signals as its sole reactivity primitive, following the TC39 Signals proposal. Signals are values that can be read, written, and observed for changes.

signal

Creates a writable reactive value:

import { signal } from '@noopjs/signals';

const count = signal(0);
count.get();         // 0
count.set(1);
count.get();         // 1

computed

Creates a derived value that re-evaluates lazily when dependencies change:

import { signal, computed } from '@noopjs/signals';

const count = signal(3);
const doubled = computed(() => count.get() * 2);
count.set(5);
doubled.get();       // 10 (re-evaluated lazily)

effect

Runs a function whenever its signal dependencies change. Returns a disposer function:

import { signal, effect } from '@noopjs/signals';

const count = signal(0);
const dispose = effect(() => {
  console.log('Count:', count.get());
});
count.set(1);        // logs: Count: 1
dispose();           // stops the effect

batch

Multiple signal writes within a batch trigger effects only once:

import { signal, effect, batch } from '@noopjs/signals';

const a = signal(1);
const b = signal(2);

effect(() => console.log(a.get() + b.get()));

batch(() => {
  a.set(10);
  b.set(20);
});                  // logs: 30 (once, not twice)

untrack

Reads a signal without creating a dependency:

import { signal, effect, untrack } from '@noopjs/signals';

const x = signal(1);
const y = signal(2);

effect(() => {
  console.log(x.get(), untrack(() => y.get()));
});                  // re-runs only when x changes

readonly

Exposes a signal as read-only (only .get()):

import { signal, readonly } from '@noopjs/signals';

const counter = signal(0);
const readOnly = readonly(counter);
readOnly.get();      // 0
readOnly.set();      // throws

Components

NoopJS components are plain functions that return DOM nodes. They are compiled at build time — the source you write is never shipped as-is to the browser.

Basic Component

// src/greeting.noop.tsx
export default function Greeting(props: { name: string }) {
  return <div>Hello, {props.name}!</div>;
}

Composing Components

Components render other components by calling them directly:

import { signal } from '@noopjs/signals';
import Greeting from './greeting.noop';

export default function App() {
  const name = signal('World');
  return (
    <div>
      <Greeting name={name.get()} />
      <input value={name.get()} onInput={(e) => name.set(e.target.value)} />
    </div>
  );
}

Async Components

Components can be async. NoopJS handles this during SSR and client rendering:

export default async function AsyncPage(props: { slug: string }) {
  const data = await fetchData(props.slug);
  return <div>{data.title}</div>;
}

Dynamic Values

When a signal is used in JSX, the DOM updates automatically:

const count = signal(0);
return <p>Count: {count}</p>;

// The compiler generates:
// const txt = document.createTextNode('Count: ');
// bindText(txt, () => 'Count: ' + count.get());
Components must be default exports of .noop.tsx files. Named exports are not yet supported for components.

NoopCSS — Atomic CSS System

NoopCSS extracts static style objects into atomic CSS classes at build time. Each CSS property-value pair becomes a single, deterministic class. Zero runtime overhead — no CSS-in-JS execution on the client.

Defining Styles

export const styles = {
  container: {
    background: '#1a1a2e',
    padding: '16px',
    borderRadius: '8px',
    display: 'flex',
    gap: '12px',
  },
  title: {
    fontSize: '24px',
    fontWeight: 700,
    color: '#e6edf3',
  },
  button: {
    padding: '8px 16px',
    cursor: 'pointer',
  },
};

Using Styles

export default function Card() {
  return (
    <div className={styles.container}>
      <h2 className={styles.title}>Hello</h2>
      <button className={styles.button}>Click</button>
    </div>
  );
}

The extractStyles function in @noopjs/css processes the source at build time:

  • Each property: value pair is hashed to a short class name (e.g., _a3f8b2).
  • The export const styles block is removed from the output.
  • A CSS file is generated with the atomic rules.
  • References to styles.x in JSX className are replaced with the generated class string.

Numeric Values

Numeric values automatically get px appended:

{ padding: 16 }        // → padding: 16px
{ fontSize: 24 }       // → font-size: 24px
{ gap: 8 }             // → gap: 8px

Inline Styles

Dynamic inline styles (with signals or expressions) use bindStyle at runtime. Static inline styles (all literal values) are extracted to atomic classes at build time. 

// Static — compiled to atomic CSS
<div style={{ color: 'red', padding: 8 }} />

// Dynamic — uses bindStyle at runtime
<div style={() => ({ color: active.get() ? 'green' : 'red' })} />

Event Handlers

Event handlers in NoopJS use event delegation. The onClick/onInput/etc. props register handlers via bindEvent, which sets a data-noop-ev attribute and delegates through a global listener. 

export default function Counter() {
  const count = signal(0);
  return (
    <div>
      <p>{count}</p>
      <button onClick={() => count.set(count.get() + 1)}>+</button>
      <button onClick={() => count.set(count.get() - 1)}>-</button>
    </div>
  );
}

Handler Extraction (Code Splitting)

When extractHandlers: true (default in production builds), each event handler is extracted into its own JavaScript chunk. Handlers are loaded lazily on first interaction. Pages with zero handlers ship zero JavaScript.

The first click on a handler-triggering element fetches the handler chunk, then the handler fires. Subsequent clicks are instant — the chunk is cached.

Available Events

All standard DOM events are supported via on{Event} props: onClick, onInput, onSubmit, onMouseEnter, onFocus, onBlur, etc.

Lists & Conditionals

Conditional Rendering

export default function Toggle() {
  const show = signal(false);
  return (
    <div>
      <button onClick={() => show.set(!show.get())}>Toggle</button>
      {show.get() ? <p>Visible</p> : null}
    </div>
  );
}

The compiler handles condition ? <A /> : <B />, expr && <JSX />, and expr || <JSX /> patterns.

List Rendering

Use .map() for simple lists:

export default function List() {
  const items = signal(['a', 'b', 'c']);
  return (
    <ul>
      {items.get().map(item => <li>{item}</li>)}
    </ul>
  );
}

The compiler replaces .map() with __noopEach() for unkeyed lists, or __noopReconcile() when a key prop is present (keyed reconciliation). This ensures efficient DOM updates when the array changes. 

// Keyed list — efficient updates when items change
{items.get().map(item => (
  <li key={item.id}>{item.name}</li>
))}

Context & Portals

Context

Share data through the component tree without prop drilling:

import { createContext, useContext } from '@noopjs/runtime';

const ThemeContext = createContext('light');

function ThemedButton() {
  const theme = useContext(ThemeContext);
  return <button className={theme === 'dark' ? 'dark' : 'light'}>Click</button>;
}

export default function App() {
  return (
    <ThemeContext.Provider value="dark">
      <ThemedButton />
    </ThemeContext.Provider>
  );
}

Portals / Teleport

Render content into a different DOM location:

import { createPortal } from '@noopjs/runtime';

export default function Modal() {
  return createPortal(
    <div className="modal">...</div>,
    document.body
  );
}

The target can be a CSS selector string, an Element, or a function returning an element.

Lifecycle Hooks

HookDescription
onMount(fn)Runs after the component is mounted to the DOM
onUpdate(deps, fn)Runs when any dependency signal changes
onUnmount(fn)Registers a cleanup function for unmount
startEffectCleanup()Starts a MutationObserver to auto-dispose effects on DOM removal
stopEffectCleanup()Stops the effect cleanup observer
import { signal, effect } from '@noopjs/signals';
import { onMount, onUnmount } from '@noopjs/runtime';

export default function Timer() {
  const elapsed = signal(0);
  let interval;

  onMount(() => {
    interval = setInterval(() => elapsed.set(elapsed.get() + 1), 1000);
  });

  onUnmount(() => {
    clearInterval(interval);
  });

  return <p>{elapsed} seconds</p>;
}

Transition Hooks

For animation on element enter/leave:

import { onEnter, onLeave } from '@noopjs/runtime';

export default function Animated() {
  return (
    <div
      onBeforeEnter={el => { el.style.opacity = '0'; }}
      onEnter={el => { el.style.opacity = '1'; }}
      onLeave={(el, done) => {
        el.style.opacity = '0';
        el.addEventListener('transitionend', done);
      }}
    >
      Content
    </div>
  );
}

Tailwind CSS Integration

NoopJS provides a first-class integration with Tailwind CSS v4. The createTailwindResolver function converts spacing-scale token values (like token.spacing.6) to their Tailwind utility equivalents (like p-6).

Setup

npm install tailwindcss @tailwindcss/vite
npm install @noopjs/vite @noopjs/compiler
// vite.config.ts
import { defineConfig } from 'vite';
import tailwindcss from '@tailwindcss/vite';
import { noopVite } from '@noopjs/vite';
import { createTailwindResolver } from '@noopjs/compiler';

export default defineConfig({
  plugins: [
    tailwindcss(),                    // Tailwind JIT
    noopVite({
      tokenResolvers: [createTailwindResolver()],
    }),
  ],
});

Usage

Use token.spacing.* and token.color.* in style objects:

import { token } from '@noopjs/runtime';

const styles = {
  card: {
    padding: token.spacing[6],   // → class includes "p-6"
    gap: token.spacing[4],       // → class includes "gap-4"
  },
  hero: {
    paddingTop: token.spacing[20], // → class includes "pt-20"
    color: token.color.blue[500],  // → class includes "text-blue-500"
  },
};

CSS Entry

Your stylesheet needs to import Tailwind and source the generated utilities:

/* src/style.css */
@import "tailwindcss";

/* Safelist any classes generated dynamically */
@source "../safelist/";

Safelist Pattern

Because Tailwind's JIT scans source files to generate CSS, dynamically-resolved utility class names (like p-6, gap-4) need to appear in scanned files. Create a safelist: 

<!-- src/safelist/index.html -->
<div class="p-6 p-4 p-8 p-12 p-16 p-20 gap-6 gap-4 gap-8 gap-12
            pt-4 pt-6 pt-8 pt-12 pt-16 pt-20 pb-4 mb-8
            text-blue-500 text-gray-300 text-gray-400"></div>
The safelist is needed because Tailwind v4 scans source files for class name strings. Since token resolution happens during NoopJS.compilation step (not in Tailwind's scan), the generated class names must exist in files Tailwind can see.

Supported Properties

createTailwindResolver handles spacing-scale properties:

  • padding, paddingLeft, paddingRight, paddingTop, paddingBottom
  • margin, marginLeft, marginRight, marginTop, marginBottom
  • gap, columnGap, rowGap
  • width, height, minWidth, maxWidth, minHeight, maxHeight
  • borderRadius
  • top, right, bottom, left
  • outlineWidth, outlineOffset
  • Color properties (color, backgroundColor, borderColor) map to Tailwind text/border/bg utilities

Custom Token Resolvers

You can create your own token resolvers for design systems:

import type { TokenResolver } from '@noopjs/compiler';

const myResolver: TokenResolver = {
  name: 'my-design-system',
  resolve(prop, value) {
    if (prop === 'color' && value.startsWith('brand-')) {
      return `text-${value}`;
    }
    return null; // fall through to other resolvers
  },
};

Custom Elements (Web Components)

NoopJS components can be exported as native Custom Elements, usable in any HTML page without NoopJS.

Creating a Custom Element

Add the @noopjs customElement directive at the top of your component file:

// @noopjs customElement 'my-counter'
import { signal } from '@noopjs/signals';

export default function MyCounter() {
  const count = signal(0);
  return (
    <div>
      <p>{count}</p>
      <button onClick={() => count.set(count.get() + 1)}>+</button>
    </div>
  );
}

The compiler generates a wrapper class that extends HTMLElement:

<my-counter></my-counter>

<script>
const el = document.querySelector('my-counter');
el.innerHTML; // Contains the component's DOM
</script>

Custom elements can be used in any framework or plain HTML.

SSR — Server-Side Rendering

NoopJS.s SSR renders components to HTML on the server, serializes the reactive graph, and sends it to the client. The client resumes the signal graph without re-running component code — true resumability, not hydration.

How It Works

  1. SSR Context — A fake DOM (ServerDocument/ServerElement/ServerTextNode) replaces the real document.
  2. Render — The component function runs, creating DOM nodes on the shim. Runtime functions detect SSR mode and record bindings instead of creating effects.
  3. Serialize — The render result includes: HTML string, signal state values, binding descriptors, and handler metadata.
  4. Client Resume — The @noopjs/client script reads the serialized state, recreates signals at their SSR values, re-attaches bindings via effects, and sets up event delegation.

SSR — Setup

Server Entry

// src/entry-server.ts
import { renderToString } from '@noopjs/server';
import Counter from './counter.noop';

export async function render() {
  const { html, state } = await renderToString(Counter);
  return { html, state };
}

Full Server Example (Node.js + Express)

// server.ts
import express from 'express';
import { createExpressMiddleware } from '@noopjs/server';
import { noopVite } from '@noopjs/vite';
import { createServer } from 'vite';
import Counter from './src/counter.noop';

const app = express();

// Dev: use Vite middleware for HMR + compilation
const vite = await createServer({
  server: { middlewareMode: true },
  plugins: [noopVite()],
});
app.use(vite.middlewares);

// SSR route
app.get('/', async (req, res) => {
  const { html, state } = await renderToString(Counter);
  res.send(`
    <!DOCTYPE html>
    <html>
    <head><title>NoopJS SSR</title></head>
    <body>
      <div id="root">${html}</div>
      <script id="__NOOP_STATE__" type="application/json">
        ${JSON.stringify(state)}
      </script>
      <script type="module" src="/src/main.ts"></script>
    </body>
    </html>
  `);
});

app.listen(3000);

Client Entry

// src/main.ts
import '@noopjs/client';   // auto-invokes init() on DOMContentLoaded

The @noopjs/client import automatically reads __NOOP_STATE__ from the DOM, recreates signals, attaches bindings, and sets up the SPA router.

SSR — Routing

NoopJS supports file-based routing via @noopjs/server's buildRoutes utility, plus client-side SPA navigation via @noopjs/client's navigate.

Programmatic Routes

For full control, define routes in your server entry:

import { matchRoute, buildRoutes } from '@noopjs/server';

const routes = buildRoutes(path.resolve(__dirname, 'src/routes'));

app.get('/*', async (req, res) => {
  const match = matchRoute(req.path, routes);
  if (!match) { res.status(404).send('Not found'); return; }

  const { route, params } = match;
  const mod = await import(route.filePath);
  const { html, state } = await renderToString(mod.default, params);
  res.send(fullHTML(html, state));
});

Client-Side Navigation

@noopjs/client intercepts <a> clicks, fetches JSON from the server (with X-Noop-Navigate: 1 header), and swaps the DOM using document.startViewTransition when available. 

// Manual navigation (e.g., from a button)
import { navigate } from '@noopjs/client';

navigate('/about');  // fetches, swaps DOM, pushes history

Prefetching

Links within viewport are automatically prefetched via prefetchUrl:

import { prefetchUrl, resetPrefetchCache } from '@noopjs/client';

// Prefetch a page manually
prefetchUrl('/blog/hello-noop');

SSR — Streaming

For larger pages, NoopJS supports streaming SSR with Suspense boundaries. The HTML shell is sent immediately, and content is streamed as it resolves. 

import { renderToStream } from '@noopjs/server';
import { Suspense } from '@noopjs/runtime';

export default function Page() {
  return (
    <div>
      <h1>My Page</h1>
      <Suspense fallback={<div>Loading...</div>}>
        {() => fetchSlowData().then(data => <div>{data}</div>)}
      </Suspense>
    </div>
  );
}

// Server sends headers + shell immediately, then streams content
const stream = await renderToStream(Page);
stream.pipeTo(new WritableStream({ write(chunk) { res.write(chunk); } }));

Caching

import { cacheRender } from '@noopjs/server';

const result = cacheRender({ ttl: 60000, key: 'homepage' }, () =>
  renderToString(HomePage)
);

Vite Plugin ( @noopjs/vite )

The Vite plugin is the primary build tool integration. It compiles .noop.tsx files, extracts CSS, and handles handler code splitting.

Configuration

import { defineConfig } from 'vite';
import { noopVite } from '@noopjs/vite';
import { createTailwindResolver } from '@noopjs/compiler';

export default defineConfig({
  plugins: [
    noopVite({
      cssMode: 'atomic',           // 'atomic' | 'inline' (default: 'atomic')
      extractHandlers: true,       // extract handlers to separate chunks (default: true in build)
      tokenResolvers: [            // optional token resolvers (Tailwind, custom)
        createTailwindResolver(),
      ],
    }),
  ],
});

CSS Modes

ModeDescription
atomic (default)Runs extractStyles() to convert export const styles into atomic CSS classes, then compiles. CSS is served via virtual module.
inlineCompiles without CSS extraction. Inline style={{...}} objects are collected and returned as CSS.

How It Works

  1. resolveId — Resolves virtual CSS modules (\0noop-css:) and handler modules (\0noop-handler:).
  2. transform — Matches *.noop.tsx files, compiles them, extracts CSS in atomic mode, registers handlers.
  3. load — Returns virtual CSS content and handler chunk code.
  4. renderChunk — Emits preload headers for handler chunks.
  5. handleHotUpdate — Forces full re-load for .noop.* files on change.

Compiler API

The compiler is used internally by the Vite plugin but can also be used standalone. 

import { compile } from '@noopjs/compiler';

const result = compile(source, {
  filename: 'counter.noop.tsx',
  extractHandlers: true,
  sourceMaps: true,
  tokenResolvers: [createTailwindResolver()],
});

console.log(result.code);     // compiled JavaScript
console.log(result.css);      // collected CSS
console.log(result.bindings); // SSR binding descriptors
console.log(result.handlers); // extracted handler descriptors

CompileOptions

filename?: string
Source filename (used for source maps and component IDs)
extractHandlers?: boolean
If true, event handlers are extracted to separate modules for code splitting
sourceMaps?: boolean
Generate source maps for the compiled output
tokenResolvers?: TokenResolver[]
Array of resolvers for design token values (e.g., Tailwind)

CompileResult

code: string
Transformed JavaScript output
map?: object
Source map (if sourceMaps: true)
bindings: BindingDescriptor[]
Signal-to-DOM binding descriptors for SSR resumption
componentId: string
Unique component identifier
css?: string
Collected CSS from inline style objects
handlers?: HandlerDescriptor[]
Extracted event handler metadata
customElementTag?: string
Custom element tag name (if @noopjs customElement directive is present)

TokenResolver Interface

interface TokenResolver {
  name: string;
  resolve(prop: string, value: string): string | null;
}

Return a class name string to replace the token value, or null to let the next resolver (or default NoopCSS handling) process it.
